## 1. Deadlocks

### What is a Deadlock?
A **deadlock** occurs when a waiting process can never change its state because other processes hold the resources it needs, and vice versa. All involved processes are permanently blocked.

### Four Necessary Conditions (all must hold simultaneously)
1. **Mutual Exclusion** — at least one resource is held in a non-shareable mode
2. **Hold and Wait** — a process holds at least one resource and waits for additional ones
3. **No Preemption** — resources cannot be forcibly taken away from a process
4. **Circular Wait** — a circular chain of processes exists, each waiting for a resource held by the next

### Resource Allocation Graph (RAG)
- Visual representation of processes, resources, and their relationships
- **Request edge**: Process → Resource (process is waiting for resource)
- **Assignment edge**: Resource → Process (resource is allocated to process)
- **Graph reduction**: Remove edges of processes that can finish (all requests satisfiable). If the graph **cannot be fully reduced**, a deadlock exists.
- **Drawback of RAG**: A process can only request **one resource unit at a time**

### Banker's Algorithm (Deadlock Avoidance)
Used to determine if a system is in a **safe state**.

**Key matrices:**
- **Max** — maximum demand of each process
- **Allocation** — currently allocated resources
- **Need** = Max − Allocation
- **Available** = Total resources − sum of all Allocations

**Algorithm steps:**
1. Compute `Need` and `Available`
2. Find a process whose `Need ≤ Available`
3. Simulate completing that process: `Available += Allocation[i]`
4. Repeat until all processes finish (**safe**) or no eligible process is found (**unsafe**)

**Example from quiz:**
- Resources: A=14, B=12, C=7, D=8
- After computing, only P1 and P2 could complete before the system ran out of available resources → **Unsafe state** (P0, P3, P4 could not proceed)

## 2. Memory Management

### Address Binding
The process of mapping symbolic addresses to physical memory addresses. Three types:

| Timing | Name | Description |
|---|---|---|
| Compile time | **Static** binding | Address known at compile time; code must be recompiled if location changes |
| Load time | Load-time binding | Final binding happens when program is loaded |
| Execution time | **Dynamic** (run-time) binding | Binding delayed until execution; requires hardware support (e.g., base register) |

> **Exam key**: Compiler binds symbolic → memory address at compile time = **Static** binding

### Memory Hierarchy
- **Registers** → accessible within **one CPU clock cycle** (True)
- Cache → SRAM, faster than main memory
- Main Memory → DRAM
- Secondary Storage → Disk/SSD

### Logical vs Physical Addresses
- **Logical address** — generated by the CPU (also called virtual address)
- **Physical address** — the actual address in main memory
- The **MMU (Memory Management Unit)** translates logical → physical

> **Exam key**: An address generated by the CPU for accessing instructions = **Logical Address**

### Dynamic Loading
- With **dynamic** loading, a routine/function is **not loaded into memory until it is called**
- Saves memory — unused routines never loaded

### Base and Limit Register Protection Scheme
- **Limit register** — holds the size of the process's address space
- **Base register** — holds the starting physical address
- CPU generates a logical address → if `logical address < limit`, then `physical address = base + logical address`; otherwise → **trap (addressing error)**

**Example from quiz (Limit=2000, Base=4000):**
- Logical address range: **0 to 1999** (2000 instructions)
- Logical address ≥ 2000 → causes a **trap**
- If logical address = 1000, base = 4000 → physical = 4000 − 1000 = **3000** (base register value is 3000 in this case)
- Physical address range: **3000–4999**

### Memory Fragmentation
- **External fragmentation** — scattered free holes in memory too small individually for new processes (caused by allocation/deallocation of processes)
- **Internal fragmentation** — allocated memory is slightly larger than requested; wasted space within partitions

> **Exam key**: Scattered holes that cannot be used for large processes = **External** fragmentation

### Dynamic Memory Allocation Strategies
Given holes: 530KB, 440KB, 180KB, 600KB, 220KB
Processes: 160KB, 570KB, 450KB, 390KB, 190KB

**First-Fit** — allocate the **first** hole that is big enough
- Scans from beginning, places process in first sufficient hole
- Total remaining holes: **660KB**

**Best-Fit** — allocate the **smallest** hole that is big enough
- Searches all holes, picks the tightest fit
- Total remaining holes: **210KB** ← **Best performance**

**Worst-Fit** — allocate the **largest** hole
- Searches all holes, picks the biggest one
- Total remaining holes: **780KB**

> **Ranking** (best to worst): Best-Fit (210KB) > First-Fit (660KB) > Worst-Fit (780KB)

## 3. Virtual Memory & Paging

### Paging Basics
- Physical memory is divided into fixed-size **frames**
- Logical memory is divided into same-size **pages**
- A **page table** maps page numbers → frame numbers
- Eliminates external fragmentation

### Page Table
- Each row: `Page Number | Valid/Invalid bit | Frame Number`
- **Valid (v)** — page is currently in main memory
- **Invalid (i)** — page is NOT in main memory (access causes a **page fault**)
- The page table **lists the page frame numbers of pages loaded into memory**

### Virtual Address Translation
Given a virtual address in hex, split into **page number** and **offset**:
- Page number → look up frame number in page table
- Physical address = (frame number × page size) + offset

**Example from quiz (page size = 256 bytes, i.e., 2 hex digits for offset):**
- `311₁₆` → page 3, offset 11₁₆ → frame B → physical = B×16 + 11₁₆ = **149₁₀**
- `E07₁₆` → page E, offset 07₁₆ → frame 8 → physical = **103₁₀**
- `F8C₁₆` → page F, offset 8C₁₆ → frame C → physical = **284₁₀**

### Page Table Size Calculation
- **32-bit virtual address**, **4KB (2¹² byte) page size**
- Number of pages = 2³² / 2¹² = **2²⁰ = 1M rows** in the page table
- Number of pages in VM = number of rows in page table

### Page Faults & Page Fault Handling Routine (PFHR)
When CPU accesses a page marked **invalid**:
1. CPU generates a page fault (trap to OS)
2. OS finds the needed page on disk (backing store / VM)
3. OS loads the page into a **free frame** in main memory
4. OS **updates the page table** (set frame number, mark valid)
5. Restart the interrupted instruction

> If there are **free frames available**, no page replacement algorithm is needed

### Page Replacement Algorithms
Used when **no free frames** are available — must evict a page.

Reference string example: `2 1 3 4 2 5 1 2 5 3 1 2 5 1 3` with **3 frames**

#### Optimal (OPT)
- Replace the page that will **not be used for the longest time** in the future
- Theoretical best — not implementable in practice (requires future knowledge)
- Result: **4 page faults** resolved with replacement

#### FIFO (First-In-First-Out)
- Replace the **oldest** page in memory
- Simple but can suffer from **Belady's anomaly** (more frames → more faults)
- Result: **9 page faults** resolved with replacement

#### LRU (Least Recently Used)
- Replace the page that has **not been used for the longest time** in the past
- Good approximation of optimal
- Result: **9 page faults** resolved with replacement

> **Performance ranking**: Optimal (4) > FIFO = LRU (9 each, in this case)

## 4. I/O Systems

### Device Driver
- The installed **device driver** helps the OS identify an I/O device during a system call
- It is the software interface between the OS and the hardware device controller

### Types of I/O Operations

#### Programmed I/O (Direct I/O)
- CPU directly controls the I/O operation
- CPU **busy-waits** (polls) for the I/O device status
- Data passes through the CPU between memory and device
- Architecture: Memory ↔ **CPU** ↔ Peripherals (CPU is in the middle)

#### Interrupt-Driven I/O
- CPU issues I/O command, then **continues other work**
- When I/O completes, the I/O controller sends an **IRQ** (interrupt request) to the CPU
- CPU stops current execution and runs the **ISR** (Interrupt Service Routine)
- ISR handles the I/O completion (checks status: Ready or Error)
- More efficient than programmed I/O — CPU not idle during transfer

#### Direct Memory Access (DMA)
- Best for **large data transfers**
- Data transfers directly between I/O device and memory **without CPU intervention**
- Architecture: Memory ↔ Peripherals directly via system bus

**DMA operation steps:**
1. DMA controller collects I/O device status, sends **DRQ** (DMA Request) to CPU
2. CPU accepts DRQ, sends **DACK** (DMA Acknowledgement) back
3. DMA controller performs **cycle stealing** — borrows the system bus (address bus, data bus, control bus) from the CPU
4. DMA controller places memory address on address bus; data transfers directly between device and memory
5. When done, DMA controller cancels DRQ, CPU cancels DACK → DMA operation ends

### Interrupts — Key Statements
- A **system call** is a **software interrupt** that a user application uses to initiate an I/O operation → **True**
- An I/O controller request is a **hardware interrupt** to the CPU resulting from a system call → **True**
- A page fault is a **user-maskable** interrupt → **False** (page faults are non-maskable / handled by OS)

## 5. Quick-Fire Recall (Flash Card Style)

- Deadlock = processes blocked forever waiting on each other
- RAG drawback = can only request one resource at a time
- Banker's Algorithm → checks for safe state via simulated execution
- Static binding = compile-time address assignment
- Registers = 1 CPU cycle access
- CPU-generated address = Logical Address
- Dynamic loading = load only when called
- External fragmentation = scattered unusable holes
- Best-Fit = least wasted space
- Page table v/i bit: v = in memory, i = not in memory (page fault if accessed)
- Page fault → PFHR loads page, updates table
- OPT replacement = replace page used farthest in future
- FIFO replacement = replace oldest page
- LRU replacement = replace least recently used page
- Device driver = software that lets OS talk to hardware
- Interrupt-driven I/O = CPU free while I/O happens, IRQ signals completion
- DMA = large transfers, no CPU involvement, cycle stealing
- System call = software interrupt → True
- Page fault = user-maskable → **False**
